Perform WiFi attacks to test network security and device behavior. This includes Karma attacks that respond to device probe requests with fake networks.

Note: Only test on networks you own or have explicit permission to test. Unauthorized attacks on networks are illegal in most jurisdictions.

Karma Attack

The Karma attack responds to probe requests from nearby devices by creating fake access points with the SSIDs that devices are searching for.

How Karma works

Automatic SSID Learning

  • Listens for probe requests on all WiFi channels
  • When a device probes for a network, the SSID is cached
  • Cached SSIDs are used to create fake access points
  • Maximum of 64 SSIDs can be cached

SSID Rotation

  • If multiple SSIDs are cached, rotates between them every 5 seconds
  • Increases the chance of catching devices looking for different networks
  • Each rotation creates a new fake AP with the current SSID

Evil Portal Integration

  • Each fake AP automatically starts an evil portal
  • Users who connect see a captive portal page
  • Credentials and other data can be captured

Beacon Broadcasting

  • Broadcasts beacon frames for all cached SSIDs every 500ms
  • Makes the fake networks visible to nearby devices
  • Devices scanning for WiFi will see these networks

Starting Karma

On-device UI

  1. Open Menu → WiFi → Attacks. You should see the attack options.
  2. Choose Start Karma Attack for automatic SSID learning, or Start Karma Attack (Custom SSIDs) to specify networks manually. The device will start listening for probe requests and broadcasting fake networks.
  3. To stop, select Stop Karma Attack. The device will shut down all fake networks.

Command line

Start Karma with automatic SSID learning

karma start

Start Karma with specific SSIDs

karma start SSID1 SSID2 SSID3

Example: karma start FreeWiFi Starbucks McDonald's

Stop Karma

karma stop

Verifying Karma is working

Look for these log messages in the terminal:

[KARMA] Received probe request from STA XX:XX:XX:XX:XX:XX for SSID 'NetworkName'
Karma cached SSID: NetworkName
Karma rotating to SSID: NetworkName
[KARMA] Evil portal started for SSID: NetworkName

From another device, scan for WiFi networks. You should see the SSIDs that Karma is broadcasting as open networks.

Testing Karma safely

  • Use your own devices for testing
  • Test in controlled environments (your home, lab)
  • Never test on public networks or other people’s devices
  • Stop the attack when testing is complete

To generate probe requests for testing:

  1. Forget a known network on your phone’s WiFi settings
  2. Turn WiFi off and on — your phone will probe for the forgotten network
  3. Walk around with WiFi enabled — phones constantly probe for known networks
  4. Use IoT devices that automatically probe for specific networks

Troubleshooting

  • Karma not detecting probes: Ensure WiFi is enabled on test devices. Try moving closer to the ESP32. Verify devices are actually probing by forgetting a known network.
  • Fake APs not appearing: Check terminal for error messages. Try restarting Karma with karma stop then karma start. Verify the ESP32 has sufficient power.
  • No devices connecting: Ensure evil portal is working correctly. Check that fake APs are visible in WiFi scans. Verify SSIDs match what devices are looking for.

Notes

  • Karma is not available on ESP32-S2 devices.
  • Karma works best when combined with evil portal functionality.
  • The device can cache up to 64 SSIDs simultaneously.
  • SSID rotation happens every 5 seconds when multiple SSIDs are cached.
